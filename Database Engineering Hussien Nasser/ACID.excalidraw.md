Isolation :
    * can my inflight transactions see changes made by other transactions ?
    ^7c5rhoue

ISSUE :: Read Phenomena ^BclHuoNa

Solution :: Isolation Levels For Inflight Transactions.

=> Read Uncommitted : No Isolation , any change from outside is visible to the Transaction, committed or not.
         (No database support except SQL Server)

=> Read Committed : each query in a transaction only sees committed changes by other transactions.
          (default isolation level for many Db Engines)

=> Repeatable Read : the transaction will make sure that when a query reads a row in first time then try to read it again it must 
        get the same value not effected by other committed transactions.


=> Snapshot : each query in the transaction only sees changes that have been committed up to the start of the transactions.
        it's like a snapshot version of the Db at this moment

=> Serializable : Transactions are run as if they serialized one after the other no Parallelism here for transaction 
        and the question here will be about how to order what run first (slows things down vs snapshot)





Database Implementation of Isolation

=>> NOTE :: Each DBMS implements isolation level differently 
=>> Pessimistic : Row Level Locks , table Locks , Page Locks to avoid lost updates ((Expensive)) 
=>> Optimistic : No Locks , just track if things changed and fail the transaction if so ((preffered over pessimistic))                =>> Pessimistic : Row Level Locks , table Locks , Page Locks to avoid lost updates  
=>> repeatable read "locks" the rows it reads but it could be expensive if you read a lot of rows 
    Postgres implements RR as snapshot, 
        so that is why you don't get phantom reads with Postgres in repeatable reads Isolation level 
=>> serializable are Usually implemented with optimistic concurrency control,
        you can implement it pessimistically with select For Update ^aVwCFucj
